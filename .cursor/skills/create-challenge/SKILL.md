# Create a challenge

Use this skill when the user asks you to create a new PsyNetSkills challenge.
The user should be able to describe the desired participant experience in prose;
you are responsible for turning that description into the repository format.

Default introduction: I will help turn a prose challenge idea into a dashboard-ready PsyNetSkills challenge with clear instructions, optional criteria, and verified references.

## Workflow

1. Anchor the intended fidelity before drafting. Ask the user to choose:
   - `fidelity`: `exact` if the attempt agent must faithfully replicate the
     proposed experiment, or `flexible` if the attempt agent may make reasonable
     implementation choices where the brief is underspecified.
   - Source of truth: user prose, a paper/reference, an existing experiment, or a
     combination.
   - Ambiguity policy: ask before deciding, propose options for approval, or use
     conservative PsyNet-native judgment.
   - Allowed adaptations: none beyond PsyNet feasibility, placeholders for
     missing assets, shorter local testing modes, accessibility/training pages,
     or other explicitly approved changes.
2. If the user clearly asks for an exact challenge, set `fidelity: exact`. Be
   cautious: whenever a detail is unclear, ask a short questionnaire or selection
   list and let the user select the true option or propose the intended answer.
   Do not silently replace the user's design with a cleaner or more conventional
   one.
3. If the user allows design discretion, set `fidelity: flexible` and record the
   main degrees of freedom in the public instructions.
4. Choose a concise lowercase slug using hyphens, unless the user provided one.
5. Use the `identify-author` skill before writing metadata.
6. Create `challenges/<slug>/INSTRUCTIONS.md` with YAML frontmatter containing
   `title`, `type`, `difficulty`, `authors`, and `fidelity`.
7. Turn the user's input into more formal prose. When the instructions are vague,
   ask them about potential clarifications. We want a moderate level of detail
   similar to what one might see in the Procedure section in a psychology research article.
8. Near the top of `INSTRUCTIONS.md`, include a short `## Fidelity` section that
   states the confirmed fidelity mode, source of truth, ambiguity policy, and
   allowed adaptations.
9. Ask the user if they want to provide any secret evaluation criteria.
   If provided, these should go into `challenges/<slug>/CRITERIA.md`. Do NOT autopopulate this with inferred content, this should only be provided by the user (and neatened up by the agent).
10. Confirm where supporting reference materials should come from. Ask first
   whether the user wants to provide materials, and ask for approval before
   searching online. Verify any paper materials against real sources; do not
   invent citations, DOIs, URLs, summaries, or bibliographic details.
11. Put any supporting public material in `challenges/<slug>/references/`.
12. Create `challenges/<slug>/attempts/.gitkeep`.
13. Run `uv run psynetsk-validate` and the narrowest useful additional checks.
14. Show the resulting text to the user and ask them to approve it; iterate if necessary.

## Challenge folder structure

```text
challenges/<slug>/
├── INSTRUCTIONS.md    # Required: instructions with title, type, difficulty, authors, and fidelity in YAML frontmatter
├── CRITERIA.md        # Optional: user-provided secret evaluation criteria
├── references/        # Optional: verified supporting materials
└── attempts/          # Required: autogenerated records of attempts made
```

## Rules

- Do not require the user to specify file names, frontmatter, or directory
  structure.
- Do not read or copy previous attempts as examples for a new challenge.
- Keep custom or real service credentials out of instructions, references, and
  criteria.
- Prefer one clear challenge over a broad bundle of loosely related tasks.
