# Participant quality telemetry report

## Experiment

`code/quality_telemetry_text/` contains a local-only PsyNet experiment with a
brief text-heavy participant flow:

1. a short introduction explaining the writing task and manual-review telemetry;
2. two open-text scenario responses;
3. one comprehension/attention check asking for the word `blue`; and
4. a closing page.

The participant-facing wording says timing and interaction summaries support
manual data-quality review. It does not threaten automatic rejection or claim to
detect AI use, bots, fraud, or dishonest behavior.

## How the telemetry skill informed the implementation

The implementation follows `psynet-participant-quality-telemetry` by separating
review-worthy in-study signals from proof claims. It records aggregate browser
signals useful for human inspection while avoiding raw keystrokes, clipboard
contents, hidden browser data, production recruiter identifiers, or private
participant records.

## Recorded fields

The browser path stores telemetry in each PsyNet page response under
`metadata.quality_telemetry` and in the structured answer payload. Fields include:

- trial id, stimulus kind, stimulus text, and participant/profile label;
- response text and attention-check pass/fail;
- page start time, submit time, response latency, and first-key latency;
- paste count and pasted character count;
- focus, blur, and visibility-hidden counts;
- keydown count, edit count, max text length, and inter-key interval summaries.

The local fixture at `evidence/psynet_quality_fixture.json` mirrors this PsyNet
response shape. Its telemetry includes `directly_recorded: false` and a
`simulation_note` so reviewers can distinguish generated evidence from a live
browser export.

## Simulated profiles

- `attentive_human_like`: typed original responses, normal latencies, no paste
  activity, no focus loss, and a passed check.
- `rushed_failed_check`: very short latencies, sparse keydown telemetry, one
  focus/visibility loss, short answers, and a failed check.
- `pasted_ai_disclosed`: disclosed AI assistance in fixture metadata, pasted
  long responses, focus/visibility loss, and sparse keydown telemetry relative
  to final text length.

All profiles are local fixtures generated by
`python3 generate_fixture_data.py`; no real participants, private data,
external services, or production credentials are used.

## Export and review process

From `code/quality_telemetry_text/`:

- `python3 generate_fixture_data.py` writes
  `../../evidence/psynet_quality_fixture.json`.
- `python3 review_quality.py ../../evidence/psynet_quality_fixture.json --output ../../evidence/manual_review.json`
  writes participant-level manual-review output.
- `python3 -m zipfile -c ../../evidence/data.zip ../../evidence/psynet_quality_fixture.json ../../evidence/manual_review.json`
  packages the exported fixture and review JSON.

## Manual-review rules

The review script flags participants for human inspection when it sees failed
checks, very fast normal responses, short normal responses, paste activity,
focus or visibility loss, sparse keydown telemetry, short first-key latency, or
declared AI assistance. The output lists each participant, observed signals,
and concrete evidence strings.

The script is intentionally conservative: `flag_for_manual_review` is not an
automatic rejection decision. It does not claim to prove AI use, bot use,
browser automation, platform fraud, or dishonest behavior.

## Evidence summary

`evidence/manual_review.log` shows that the attentive fixture is unflagged, while
the rushed and pasted/disclosed fixtures are flagged with interpretable reasons.
`evidence/data.zip` contains the generated PsyNet-format fixture and review JSON.
`evidence/participant.mp4` is a screen-only recording because the experiment has
no participant-facing audio stimuli.
